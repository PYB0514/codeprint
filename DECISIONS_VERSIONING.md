# 버전 관리 체계 결정

## 문제
기능을 추가할 때마다 버전을 어떻게 붙일지 기준이 없었다.
단순히 v1.1, v1.2로 가면 주요/마이너 구분이 안 되고,
v1.9 다음이 v2.0이 되면 너무 쉽게 메이저가 올라가는 문제도 있었다.

## 결정 과정

### 1차 시도: v1.x 단일 체계
- v1.1, v1.2 ... v1.9 → v2.0
- **탈락 이유**: 9번만 업데이트하면 v2.0이 되어버림. 메이저 버전의 의미가 희석됨.

### 2차 시도: v1.x vs v1.x01 (마이너 구분)
- 주요: v1.5 → v1.6
- 마이너: v1.5 → v1.501 → v1.502
- **탈락 이유**: v1.7 다음 마이너가 v1.701이 아니라 v1.501로 흘러갈 혼동 가능성.

### 3차 시도: v1.x.00n 3단계 체계 (확정)
```
v{메이저}.{주요}.{마이너}

v1.6        ← 주요 기능 추가
v1.6.001    ← 해당 버전 내 마이너 업데이트
v1.7        ← 다음 주요 기능
v1.10, v1.11 ... ← 주요 버전은 제한 없이 증가
v2.0        ← 서비스 전면 개편 수준일 때만
```

## 결과

**주요 버전 (v1.x) 기준 — 아래 중 하나라도 해당:**
- 새 기능 카테고리 추가 (언어 지원, 결제, 커뮤니티, AI 등)
- 사용자가 "새로 생겼네" 하고 체감하는 기능
- 인증·배포·인프라 변경
- DB 스키마 변경 (Flyway 마이그레이션 포함)

**마이너 (v1.x.00n) 기준:**
- 기존 기능 안에서의 개선: UI 수정, 버그 수정, 부가 페이지, 설정 변경

**메이저 (v2.0) 기준:**
- AI 전면 도입, 완전 리디자인 등 "시즌2" 수준의 전면 개편
