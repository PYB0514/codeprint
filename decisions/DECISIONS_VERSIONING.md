# 버전 관리 체계 결정

## 문제
기능을 추가할 때마다 버전을 어떻게 붙일지 기준이 없었다.
단순히 v1.1, v1.2로 가면 주요/마이너 구분이 안 되고,
v1.9 다음이 v2.0이 되면 너무 쉽게 메이저가 올라가는 문제도 있었다.

## 결정 과정

### 1차 시도: v1.x 단일 체계
- v1.1, v1.2 ... v1.9 → v2.0
- **탈락 이유**: 9번만 업데이트하면 v2.0이 되어버림. 메이저 버전의 의미가 희석됨.

### 2차 시도: v1.x vs v1.x01 (마이너 구분)
- 주요: v1.5 → v1.6
- 마이너: v1.5 → v1.501 → v1.502
- **탈락 이유**: v1.7 다음 마이너가 v1.701이 아니라 v1.501로 흘러갈 혼동 가능성.

### 3차 시도: v1.x.00n 3단계 체계 (확정)
```
v{메이저}.{주요}.{마이너}

v1.6        ← 주요 기능 추가
v1.6.001    ← 해당 버전 내 마이너 업데이트
v1.7        ← 다음 주요 기능
v1.10, v1.11 ... ← 주요 버전은 제한 없이 증가
v2.0        ← 서비스 전면 개편 수준일 때만
```

## 결과

**주요 버전 (v1.x) 기준 — 아래 중 하나라도 해당:**
- 새 기능 카테고리 추가 (언어 지원, 결제, 커뮤니티, AI 등)
- 사용자가 "새로 생겼네" 하고 체감하는 기능
- 인증·배포·인프라 변경
- DB 스키마 변경 (Flyway 마이그레이션 포함)

**마이너 (v1.x.00n) 기준:**
- 기존 기능 안에서의 개선: UI 수정, 버그 수정, 부가 페이지, 설정 변경

**메이저 (v2.0) 기준:**
- AI 전면 도입, 완전 리디자인 등 "시즌2" 수준의 전면 개편

## 현재 태그 현황 (3차 체계 기준, v1.35까지 운영)
| 태그 | 내용 |
|---|---|
| v1.5 | CI/CD + Railway/Vercel 배포 + AWS S3 + 랜딩 페이지 |
| v1.5.001 | 패치노트 페이지, 버전 체계 도입 |
| v1.6 | C#, Ruby, PHP, Swift 언어 지원 추가 (총 11개 언어) |
| ... | v1.35까지 동일 체계로 운영 |

## 4차 결정: SemVer 전환 (2026-06-08)

### 문제 인식
3차 체계(v1.x.00n)는 나름의 논리로 만든 커스텀 체계였지만, 두 가지 문제가 있었다.

첫째, **버전 단위가 PR 기준이었다.** 사용자 입장에서 서비스가 변하지 않는 `docs:`, `chore:`, `test:` PR에도 버전이 붙고 있었다. 버전이란 사용자가 체감하는 변화의 단위여야 하는데, 개발자의 작업 단위(PR)와 혼재됐다.

둘째, **현업 SaaS와 다른 방식이었다.** Vercel, Linear, Figma 같은 서비스들은 사용자에게 `v1.35` 같은 버전을 공개적으로 노출하지 않는다. 내부 배포 추적은 빌드 번호나 날짜 기반으로 하고, 사용자에게는 changelog로 변화를 전달한다.

### 결정: Semantic Versioning (SemVer) + changelog 방식

**버전 형식:** `v{MAJOR}.{MINOR}.{PATCH}`

| 단계 | 올리는 시점 |
|---|---|
| **PATCH** (0.1.0 → 0.1.1) | 버그 수정, 사용자에게 보이는 작은 수정 |
| **MINOR** (0.1.0 → 0.2.0) | 하위 호환되는 새 기능 추가 |
| **MAJOR** (0.x → 1.0.0) | 프로덕션 준비 완료 (안정적 유료 서비스) / 하위 호환 깨지는 변경 |

**버전 태그를 붙이는 기준 (사용자 체감 변화):**
- `feat:` PR — MINOR 또는 PATCH 증가
- 사용자에게 보이는 `fix:` PR — PATCH 증가
- 서비스 동작에 영향을 주는 인프라/보안 변경 — PATCH 이상

**버전 태그를 붙이지 않는 기준:**
- `docs:`, `chore:`, `test:`, `refactor:` PR — 태그 없음

**현재 버전:** `v0.1.0`에서 재시작.
SemVer 관습에서 `v1.0.0`은 "프로덕션 준비 완료"를 의미한다. 유료화 미완료·공개 안정화 전인 현재는 `v0.x` 대가 정직하다.

**탈락 이유 (3차 체계 유지):** 현업 방식과 거리가 있고, 버전의 의미(사용자 체감)가 희석돼 있었다.

### Why
직접 체계를 만들어 운영해보면서 "PR 단위 = 버전 단위"가 아니라는 걸 경험으로 인식했다. 버전은 개발자가 아닌 사용자 관점의 단위여야 한다는 원칙을 적용해 SemVer로 전환.
